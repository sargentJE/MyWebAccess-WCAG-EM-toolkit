## 0. Decisions — RESOLVED 2026-06-13

1. **Scope: full sweep.** Execute E1–E7 in order, one branch+PR per epic, paused
   for review between PRs.
2. **PDF audit scope: inventory as documents.** Build a document inventory that
   lists the ~24 live PDFs (22 newsletters, annual accounts, walk leaflet) for
   manual review — `wcagEm.technologiesReliedUpon` lists `"PDF"` and a browser
   cannot meaningfully axe-scan one. Do **not** crawl-as-pages; do **not** silently
   drop. Drives E5.
3. **Blocked/challenged pages: try to access, else manual review.** For any page
   behind the Cloudflare challenge (the `/event*` section) or otherwise blocked,
   first attempt a workaround to reach the real content; only if no workaround
   succeeds, route it to a **manual-review queue** — never report challenge markup
   as findings, never silently drop. Strategy in §0a. The execution session may
   decide the keep-vs-flag detail of the sample list, consistent with this.

## 0a. Challenge-access strategy (blocked pages) — drives E1

Ordered by robustness; implement the safe automated options behind config, document
the rest. A blocked page is only "could not audit" after these are exhausted.

1. **WAF allowlist / bypass header (robust; Jamie-actionable at MyVision).** Add a
   Cloudflare rule that skips the managed challenge for the auditor — by source IP,
   or a secret request header the scanner sends (wire through `scan`/`auth` config).
   The only option that reliably survives a 2-hour run; primary path for a Track B
   events re-run.
2. **Wait-for-auto-solve (no infra change).** A managed challenge often self-clears
   in a real browser after a few seconds and redirects to the real page. After
   detecting a challenge, optionally wait/poll up to a bounded timeout and re-check
   `pageOutcome` before giving up. Cheap; partial success.
3. **`cf_clearance` via `auth.storageState` (no infra change; fragile).** Solve the
   challenge once manually, export `cf_clearance` + `__cf_bm` into the config's
   existing `auth.storageState`. IP+UA-bound and expires in ~30 min — document,
   don't depend on it.
4. **Fallback — manual-review queue.** Still challenged at scan time →
   `pageOutcome:'challenge'`, surfaced in the scan-health block and the E7 manual
   backlog as "could not audit automatically — review by hand." Guaranteed floor
   under options 1–3.

Treat options 2 and 4 as in-scope for E1; option 1 as a config surface E1 exposes
(the rule itself is Jamie's to add); option 3 as documented-but-optional.

### E2 — R2: fair, deterministic, recursion-aware seed budget **[red-team: redesigned]**

**My original "fetch all configured leaf sitemaps, then allocate" was wrong.** A
shipped config (`configs/legacy-events-recon.json:14-19`) already seeds a bare
`/sitemap_index.xml` whose children are discovered only at runtime by the
existing recursion (`sitemap.mjs:79-84`). "Allocate across configured sitemaps"
then degenerates to one bucket. B1 (fairness) and B3 (seed the index) are
**co-dependent and must be designed together.**

**Corrected algorithm (`sitemap.mjs` `getSitemapSeeds`):**

1. **Phase 1 — expand the index tree.** BFS following only `.xml` index entries;
   classify a sitemap as a **leaf by its body** (≥1 non-`.xml` `<loc>`), _not_ by
   URL extension — the current `/\.xml/` regex (:80) mis-treats `/sitemap?page=2`
   as a page and a `.xml` content URL as a sitemap. **Bound total child-document
   fetches** (anti-amplification — today only _pages_ are capped, a malicious
   index could fan out to thousands of fetches).
2. **Phase 2 — cache** each leaf's parsed `<loc>` list (don't re-fetch).
3. **Phase 3 — deterministic round-robin reservoir** across leaves sorted by URL:
   round `r` takes each leaf's `r`-th URL until `maxUrls`. **Pure round-robin, no
   floor, no proportional** — the downstream sampler clusters by
   `pageType::firstPathSegment` and the random pool draws from the whole
   inventory, so round-robin maximizes _content-type_ representation (a 9-URL
   `give_forms` sitemap gets in before a 1001-URL events sitemap drowns it).
   Proportional re-introduces starvation.

**Determinism trap [red-team]:** do **not** parallelize the loc fetches for
speed — arrival order would make the clip set non-deterministic and break
`randomSeed` reproducibility (WCAG-EM Step 3c). `discover.mjs:237` sorts the
inventory lexicographically, which _masks_ this in `urls.txt` while the random
sample still diverges. Fetch in config order (or re-sort leaves by URL before the
round-robin) and **add a two-run byte-identical-`inventory.json` determinism
test** with a stub returning lists in shuffled order.

**Telemetry (B2):** return per-sitemap `{url,found,contributed,clipped}` +
`reachedCap` + `neverReached`; `logger.warn` on clip; add to
`inventory-metadata.json`. `reachedMaxPages:false` today is reassuring-and-wrong
(it tracks the crawl ceiling 220, not the seed cap 1200, which _was_ hit).

**B3 config:** add `sitemap_index.xml` to the MyVision config (so venue/organiser
children are discovered), and add the **missing recursion test** — `sitemap.mjs`
recursion is currently untested (no `<sitemapindex>` fixture exists). Seed the
event/venue/organiser sitemaps so they are _available_; until the §0a workaround or
WAF rule lands, event pages will surface as `challenge` → manual-review rather than
findings — that is expected, not a regression.

**Needs an ADR** (0018) — changes seeding semantics. New config field →
update `docs/guides/config-guide.md` (the `docs-config-coverage` guard fails
otherwise) and keep `configs-valid` green.

## 5. Contract-safety checklist (E1 must satisfy all before merge)

1. No `outcome` on `summary.findings[]` rows (use `pageOutcome`); assert
   `findings[*].outcome === undefined`.
2. One synthetic `pageOutcome:'challenge'` entry carrying fake violations
   produces **zero** rawFindings in `portal-export.json`, **zero**
   `pagesTested`/findings in `report-builder-draft.json`, and **zero** SC flips

3. `pageViewsScanned`, `samplePagesScanned`, `processRuns`, and `criteriaOutcomes`
   do **not** count non-`ok` views/states.
4. `validateExports:"warn"` stays clean: `portal-canonical-scan.schema.json`
   (esp. the `rawFinding.allOf` critical/serious evidence gate) and the
   `report-builder-draft.schema.json` root `additionalProperties:false` still pass.
5. Process-path Response plumbing exists (a process whose `goto` lands on a
   challenge is covered by a test).
6. `classifyPageOutcome` unit tests: normal 200 whose body contains "just a
   moment" → `ok`; thin valid page, no upstream error → `ok`; `cf-mitigated`
   403 → `challenge`; `about:blank` after a prior error → `empty`.
7. A detected challenge does not consume `scan.retries` nor land in `pagesFailed`.
